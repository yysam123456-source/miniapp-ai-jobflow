# 一、竞品分析（竞析 / Compa）

## 1. 竞品全景图
按求职链路节点分三类，代表产品如下：
- **AI 简历类（直接竞品·简历节点）**：AI简历姬、100分简历（3000+ 模板全免费）、超级简历、WonderCV、职徒简历（中英/行业适配）、知页、锤子、鹅来面 OfferGoose、52cv、Canva。
- **证件照类（直接竞品·证件照节点）**：合合照（600+ 规格、发丝级抠图、看广告得高清）、吉吉照（全免费）、和和照（签证强）、nik拼（城市回执准）、完美证件照（一键合规）、Cutout（抠图强、web）。
- **同域工具箱（形态参考）**：洛克孵蛋工具箱、作业答案来了（多工具聚合）。

**关键判断**：三类竞品均只覆盖"单节点"，无任何一款将"简历 ↔ 证件照"打通。**一站式串联 = 市场空缺，即我方核心差异化（F1/F5/F8）**。

## 3. 市场数据 & 趋势
完整市场面（CONTEXT §3）：国内 AI 小程序用户突破 **3 亿**（年增 25%）；小程序生态月活突破 **10 亿**；小程序广告市场预计 **850 亿元**；微信成长计划半年 **4.5 万**小程序（70%+ 个人开发者）；**2026 高校毕业生预计 1270 万**（就业竞争加剧）。
**趋势判断**：①微信小程序是最大增量场，但简历/证件照竞品多以 web 为主、未吃微信裂变红利；②毕业季刚性需求×就业压力，证件照+简历双刚需叠加，串联可借社交分享低成本获客；③850 亿广告盘+4.5 万小程序供给，说明流量主变现路径已被验证，我方"保体验≤1次插屏"可平衡变现与口碑。

## 4. 用户感知视角（独立补充，不重复瑞思）
- 简历类用户普遍感知"web 端好用但分享麻烦、无社交裂变"，且证件照需切到另一 App，链路断点明显。
- 证件照类用户的核心吐槽是"只产出照片、无下游简历场景"，照片价值未被延伸。
- 工具箱类证明"多工具聚合"形态被接受，但现有聚合多为娱乐/学习向，求职严肃场景的聚合仍是空白认知。
- 用户对"AI 润色是否虚构经历"敏感——我方 `sanitizeAndParseResumeJson`（禁虚构/清洗）是信任锚点。

## 5. SWOT 分析
- **S（优势）**：免费混元降本；串联/双入口/微信分享的差异化；craftisle-resume + craftisle-app 现有代码资产可快速兑现。
- **W（劣势）**：首版模板仅5套、证件照引擎"非免费额度"需独立成本评估；云函数 vs 现有后端运行时不兼容（R1/R2）。
- **O（机会）**：3亿 AI 小程序用户增量场、1270万毕业生刚性需求、市场空缺无人占据串联位。
- **T（威胁）**：合合照600+规格/发丝级抠图、100分简历3000+模板的体量压制；第三方引擎成本与合规风险。

## 6. 差异化机会（核心·呼应代码复用）
差异化不止"设计主张"，而是**有现成资产可快速兑现**，证据来自 CONTEXT §4 的 codegraph+semble 验证：
- **串联工作流 / 元素级编辑 / 导入自动映射 / 双入口 / 微信分享** → 由 `craftisle-resume` 支撑：`createResumePdfFile`(F7)、`buildDocx`(F7b)、`analyzeResumeSystemPrompt`+`sanitizeAndParseResumeJson`(F6c)、import importers(F6b)、`resumeDataSchema`+`applyResumePatches`(RFC6902, F6元素级) 均"直接/改造复用"。
- **F2/F3/F4 后处理** → 由 `craftisle-app` 支撑：`passportPhoto()`(Sharp 换底/裁剪/导出, 含 China/US/UK/EU 50+规格)、`id-photo`/`remove-bg`(发丝级抠图/换底)、`colorAdjust` 等 17+ 图片原语可直接改写进云函数。**这修正了旧结论"证件照零复用"**。
- **关键杠杆**：`AI_PROVIDERS` 含 `openai-compatible`，若混元 Hy3 暴露 OpenAI 兼容协议(R3)，则 optimize/importParse 的 LLM 层"仅改 provider 配置即可复用全部 prompt/清洗"，成本最低、风险最小。

## 7. 行动建议
1. **评估第三方开源检测引擎**（F3 优先）：对比成本/精度，并复用 craftisle-app 人脸定位实现作评估输入。
2. **确认混元兼容协议(R3, P0)**：验证 Hy3/Hy Image 3.0 是否 OpenAI 兼容，决定 optimize/importParse 复用成本。
3. **验证证件照后处理复用 craftisle-app 的可行性**：将 Sharp 逻辑移植/改写为云函数或独立图片服务，跑通 F2/F3/F4 后处理。
4. **锁定架构 A/B(R1)**：优先方案 B（resume 全链路零改造复用），证件照与微信链新建；强约束云开发则退化为 A。
5. **提前验证 @react-pdf/renderer 云函数体积/冷启动(R2)**，避免导出链路上线即翻车。

## 3. 北极星指标 + 关键漏斗
**北极星指标**：周完成「证件照生成 → 简历导入/填写 → PDF 导出 → 微信分享」闭环的用户数（Weekly Closed-Loop Users, WCLU）。
**关键漏斗与转化率目标区间**（基于行业经验假设，需灰度验证）：
| 步骤 | 环节 | 转化率目标 | 假设说明 |
|---|---|---|---|
| 1 | 首页 → 进入工作流 | 35%–50% | 清晰 CTA + 双入口（F5） |
| 2 | 进入工作流 → 生成证件照 | 60%–75% | Hy Image 3.0 生成成功率 ≥90% |
| 3 | 生成证件照 → 导入/填简历 | 55%–70% | F6b 自动映射降低摩擦 |
| 4 | 填简历 → 导出 PDF | 65%–80% | 复用 craftisle-resume 导出稳定 |
| 5 | 导出 PDF → 微信分享 | 25%–40% | 预览即分享 + 裂变钩子 |
整体端到端闭环率目标：3.5%–16.8%，首版以 8% 为基准线。

## 4. 指标定义与埋点方案
**需追踪事件**：`workflow_started`、`photo_generated`、`resume_imported`、`resume_filled`、`pdf_exported`、`docx_exported`、`shared`、`ad_shown`、`ad_clicked`、`face_detect_failed`。
**核心指标**：激活率、闭环完成率、7 日留存、分享率、单用户 AI 调用成本、流量主 ARPU/eCPM。
**埋点复用**：导出/导入事件可对齐 craftisle-resume 现有 `createResumePdfFile`/`import` 包调用点，降低埋点改造成本。

## 5. 目标设定建议（首版上线 90 天）
- DAU：第 90 天 ≥ 3 万（占毕业生 0.24%，保守）
- 周闭环完成数（WCLU）：≥ 8 万/月
- 分享带来新增：分享率 30% 下，裂变系数 K≈0.3，月新增 ≥ 2.4 万
- 7 日留存：≥ 25%
- 单用户 AI 成本：≤ 0.05 元/活跃用户（依赖混元免费额度，R3/R4）

## 6. 风险与口径
- **R5 数据割裂**：Spec 要求与 web 端 Craftisle 完全割裂，无法跨端归因。口径：小程序侧独立埋点库。
- **免费额度监控**：设告警「单日 AI 调用成本 / 免费额度占比」，超 80% 触发降级（关闭非核心 AI 如 F6c 润色）。
- **内容安全拦截率**：追踪 `content_unsafe_blocked / 总生成量`，目标 < 2%。

## 2. 核心痛点与 JTBD

## 3. 用户场景细化（Spec §4 A/B/C）
- **场景 A·纯证件照（低门槛引流）**：进入→选规格→Hy Image 3.0 生成（F2）→换底裁剪导出（F4）→合规检测（F3）→分享/保存。情绪：生成前焦虑→合格后释然→分享满意。
- **场景 B·完整求职流（主转化）**：选模板（F6）→导入旧简历 AI 自动填充（F6b）→AI 润色（F6c）→生成职业照（F2/F4）→PDF 导出（F7）→微信分享（F8）。情绪：导入即减负的惊喜→润色见长的认可→一次性闭环的强满足。
- **场景 C·双入口续接（留存抓手）**：先走 A 拍好照→退出→再从小程序"简历前置入口"（F5）进入→历史照片自动带入→续做 B。情绪：跨次无缝的信任感。

## 5. 关键用户洞察（差异化卖点）
1. **免费混元零成本权益**：对标合合照"看广告得高清"、吉吉照全免费，我们天然零边际成本，可做"免广告高清"。
2. **微信原生分享→社交裂变**：F8 原生分享，完稿直发 HR/班级群，竞品无此闭环。
3. **串联不离开微信→极低摩擦**：单点竞品逼迫切换，一站式闭环显著降低弃单率。
4. **双入口续接→降低决策压力**：F5 支持"今天只拍照、改天续简历"，契合学生碎片化时间。
5. **信息一次录入全链路复用**：照片/简历数据互通，直击"重复填信息"最大痛点。

## 6. 复用如何转化为用户可感知价值
- **更快上线**：简历全链路 F6/F6b/F6c/F7/F7b 直接/改造复用 craftisle-resume 内核——`createResumePdfFile`、`buildDocx`、`analyzeResumeSystemPrompt`+`sanitizeAndParseResumeJson`、`resumeDataSchema`、`getModel()` 的 `openai-compatible`→ 换混元仅改 provider 配置。用户感知：上线即有 5 套模板+PDF/Word 导出+STAR 润色，非从零。
- **更稳/更专业**：证件照后处理 F4 复用 craftisle-app 的 `passportPhoto`（含 1inch/China/US/UK/EU 50+ 规格）、`remove-bg`/`id-photo`、`colorAdjust` 等 17+ 原语。用户感知：换底/抠图达合合照级精度，规格覆盖教资/国考/签证。
- **免费/稳定**：复用已验证业务逻辑，减少自研 bug 与冷启动风险，配合免费混元，用户得"稳定且零成本"体验。

## 7. 风险（体验/采纳层）
- **R-UX1 首次生图等待**：Hy Image 3.0 生图耗时，A 场景开场即等待易流失 → 需生成中进度/预览占位。
- **R-UX2 合规不通过挫败**：F3 检测不过无引导重试 → 需即时修正提示。
- **R-UX3 微信登录失效**：F9 openid 登录态过期致历史丢失 → 需静默续期+本地缓存兜底。
- **R-UX4 复用鸿沟**：craftisle-app 为 Node/Sharp 服务端，无法直接入云函数，后处理移植若延迟，F4 上线质量打折 → 优先移植核心逻辑。
- **R-UX5 混元协议不确定**：若 Hy3 无 OpenAI 兼容协议，F6c/F6b 复用成本上升、润色质量波动。
